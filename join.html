Runbook Order

Follow the join flow in rows and copy exactly what you need.

This page is arranged in order: fastest start, shell baseline, manual build path, key setup, private checks, then runtime verification and common fixes.

Step 1

Fastest start

Use the Docker-assisted path first if you want the shortest path to a working local setup. It handles bootstrap/build work while leaving runtime execution on your host.

Best for First-time operators who want a clean path with the fewest moving parts.
What it does Builds runtime artifacts, validates manifests, and moves straight into the normal Atho startup flow.
What you still need Git, Python 3.9+, Docker Desktop or Docker Engine, and API auth after bootstrap.
macOS / Linux 
git clone <repo-url>
cd <repo-dir>
python3 run.py mainnet
Windows PowerShell 
git clone <repo-url>
cd <repo-dir>
py -3 run.py mainnet
Step 2

Baseline shell setup

Set your network, create the virtual environment if needed, then create API auth before normal wallet or node operations.

Networks Use mainnet, testnet, or regnet consistently in launcher, CLI, and scripts.
venv rule Keep Atho inside one isolated environment so auth, wallet, and node tools all use the same packages.
Auth rule 401 and 403 errors usually mean the auth step was skipped or stale headers are being reused.
Set network 
export ATHO_NETWORK=mainnet
$env:ATHO_NETWORK="mainnet"
Create venv 
python3 -m venv .venv
source .venv/bin/activate
python -m pip install --upgrade pip setuptools wheel
Create API auth 
./.venv/bin/python Src/Api/auth.py
Check chain info 
curl -H "X-API-Key:<KEY>" \
  -H "X-User:<USER>" \
  -H "X-Pass:<PASS>" \
  http://127.0.0.1:<full_api_port>/chain/info
Step 3

Manual build path

Use this path when you want full control over bootstrap, dependency installation, and backend startup without relying on the fastest Docker-first flow.

Use this when You want to control each build stage directly or you are working on a host where Docker is not the preferred path.
Critical step Do not skip the dependency install and build pipeline before trying backend-only launch.
Common failure Missing toolchain or stale `.venv` state causes more manual-path failures than anything else.
Clone + install 
git clone <repo-url>
cd <repo-dir>
python3 -m venv .venv
source .venv/bin/activate
python -m pip install --upgrade pip setuptools wheel
python -m pip install -r requirements.txt
Build pipeline 
python build.py --network mainnet
./.venv/bin/python Src/Falcon/Falcon/install_platform_binary.py
Backend without GUI 
python run.py mainnet --no-gui

Key Management

Run key manager before normal wallet or miner use.

Create or import signing material first, then confirm the default key and wallet security policy before you try normal send, receive, or miner flows.

Step 4

Create your first wallet record

The safest first run is a mnemonic-backed key, then an address check, then a balance check after the node and API are healthy.

Recommended order Finish bootstrap, enter the CLI wallet tool, create a mnemonic key, confirm addresses, then start normal operations.
Why first Wallet and miner flows are blocked until a valid default signing identity exists for the active network.
Lockbox option After creation, you can switch the wallet into password + AES lockbox mode with the Kyber-backed recovery path.
CLI wallet bootstrap 
source .venv/bin/activate
./.venv/bin/python Src/GUI/cliui.py

# inside the Atho CLI
wallet new 24
wallet addresses
wallet balance
Step 5

Manage default keys and imported identities

Use the GUI when you want the visual workflow. Use the CLI helpers when you need repeatable operator control over defaults, renames, imports, and deletes.

Supported actions Create, recover, import, export, rename, set default, and remove stale records per network.
Safety rule Keep the correct default key on the correct network so wallet sends and mining flows do not sign under the wrong identity.
Security stack Wallet lockbox mode uses Argon2id, AES 256 GCM, and the Kyber-backed DEK recovery path already described on the wallet and Kyber pages.
Key management CLI 
./.venv/bin/python Src/Accounts/change_default_key.py list
./.venv/bin/python Src/Accounts/change_default_key.py set-default --network <mainnet|testnet|regnet> --id miner_1
./.venv/bin/python Src/Accounts/change_default_key.py rename --network <network> --old miner_1 --new validator_main
./.venv/bin/python Src/Accounts/change_default_key.py import --network <network> --id import_1 --file /path/to/key.txt
./.venv/bin/python Src/Accounts/change_default_key.py delete --network <network> --key import_1
Private Layer Testnet Runbook

Validate public/private send paths before production deployment.

This runbook focuses on Platinum Shield behavior in testnet: prove all three send directions, confirm replay resistance, and verify note/nullifier state transitions across mempool, block inclusion, and post-apply reconciliation.

Step 6

Private testnet checklist

Run the three value-flow directions, confirm the safety assertions, then read the logs that prove the state transitions happened where they should.

Public -> Private Fund private notes from public UTXOs and confirm commitment and encrypted payload creation.
Private -> Private Spend by nullifier and confirm pending-spend to spent transitions after a block lands.
Private -> Public Release value back into public outputs and check the accounting path for parity.
Replay protection Cross-network replay attempts and duplicate nullifiers must fail in mempool and block validation.
Bundle integrity X-address and bundle-hash checks must remain aligned on the receive and validation paths.
Operator logs Watch private mempool admits, block private-tx outcomes, and post-apply reconciliation state changes.

Runtime Checks

Verify node health fast and keep the most common fixes close.

Check runtime health first, then use the copied one-line fixes for auth, venv, binaries, and stale processes before you dig deeper.

Step 7

Runtime signals to watch

Peer handshakes, sync height movement, and clean process state are the fastest proof that the environment is usable.

Peer handshake Look for handshake_ok and active peers in the network logs.
Sync progression Tip height should keep moving toward the best known chain instead of stalling immediately.
Process health Use the node stop helper to catch stale background processes and port conflicts before deeper debugging.
List active nodes 
./.venv/bin/python Src/Main/stop.py --active
Stop stale nodes 
./.venv/bin/python Src/Main/stop.py --all
Watch network log 
tail -f logs/mainnet/network/network.log
