Kyber Implementation Private Flow + Wallet Lockbox

How Atho uses Kyber on chain and at rest.

Atho uses Kyber in two separate lanes on purpose. The private payment path uses Kyber 512 to derive a per-output shared secret for note delivery, while wallet lockbox mode uses Kyber 1024 to wrap the AES data-encryption key that protects wallet secrets at rest.

Current Code Profile

  • Private layer lane PRIVATE_KYBER_LEVEL = 512, ciphertext size 768 bytes, optional view tag 4 bytes.
  • Recipient identity Private bundle bytes are version || falcon_public_key || kyber_public_key, then sha3_384 derives the bundle hash and X-address.
  • Wallet lockbox lane Lockbox mode uses kyber1024_ref by default while AES 256 GCM performs the actual wallet payload encryption.
  • Restore boundary Current deterministic bundle identity can be recreated from the same mnemonic path, but legacy random bundles still require the old Kyber secret backup.
Private Output Build Path

Kyber is the recipient side secret agreement layer for private sends.

  1. Sender parses the recipient bundle and rejects the output if the X-address does not match the bundle hash.
  2. The bundle exposes the recipient Falcon public key and the recipient Kyber public key together.
  3. Sender encapsulates to the recipient Kyber public key and gets a fresh shared secret plus Kyber ciphertext.
  4. That shared secret derives the recipient id, optional 4-byte view tag, and the AES key used for note payload encryption.
  5. The output stores the Kyber ciphertext, AES nonce, clear amount field for v5 rules, and encrypted note seed payload.
  6. Receiver later decapsulates with the matching Kyber secret key and only then recovers the note seed and message.

Current v5 Payload Layout

The active private payload format is: view_tag(4) + kyber_ciphertext(768) + aes_nonce(12) + amount_plain(8) + ciphertext_seed(...).

  • Kyber ciphertext Needed so the receiver can decapsulate and reproduce the shared secret that unlocks the note payload.
  • Amount field Kept in the product-v1 payload layout so sender, mempool, and block logic stay aligned on current private output rules.
  • ciphertext_seed AES GCM encrypts the note seed and optional memo, with the note commitment used as authenticated data.
Wallet Receive Path

Scanner logic uses Kyber for deep recovery, not just labeling.

Atho does not guess private note keys blindly. Wallet scan prefilters candidate outputs, decapsulates the Kyber ciphertext, and only accepts a note after the recovered seed regenerates the expected Falcon note key and commitment.

Fast reject layer

Bundle hash prefixes and optional view tags let the wallet skip most unrelated private outputs before full decapsulation work begins.

Performance Filter

Kyber decapsulation

The scanner tries local Kyber secret keys against the stored ciphertext. If decapsulation fails, the output is not for that wallet.

Recipient Check

Ownership confirmation

After decapsulation, the wallet checks recipient id, verifies the view tag when present, decrypts the note seed, regenerates the Falcon child key, and recomputes the commitment.

At Rest Encryption

Kyber protects the DEK recovery lane while AES protects the wallet payload.

  1. Argon2id derives the password key from the wallet password and KDF metadata.
  2. A random 32-byte DEK encrypts the wallet secret payload with AES 256 GCM.
  3. The same DEK is wrapped once by the password-derived key with AES 256 GCM.
  4. Kyber 1024 encapsulation creates a shared secret, HKDF turns that into a wrap key, and that wrap key protects the same DEK again.
  5. The Kyber secret key itself is AES wrapped under the password key so the wallet can recreate the Kyber unwrap path on unlock.

What the Lockbox Enforces

  • Kyber is mandatory Lockbox mode fails closed if Kyber is disabled. The wallet expects the PQ wrap path to exist.
  • Two DEK recovery paths Password unwrap and Kyber unwrap must reproduce the same DEK before the wallet accepts the decrypted payload.
  • Schema discipline Wallet lockbox metadata uses athokey-lockbox-v1, and advanced unlock exports use athokey-kyber-unlock-v1.
  • Role separation Kyber does not replace AES. Kyber wraps the DEK; AES encrypts the actual wallet secret payload.
Deterministic Restore

Current bundle identity can be rebuilt from the same mnemonic context.

  • Kyber seed domain The deterministic branch uses context ATHO-HD-KYBER-V1 and mixes network, account, role, index, and Kyber level into the info string.
  • Bundle recreation Same mnemonic, passphrase, and path recreate the same Kyber keypair, bundle hash, X-address, and bundle view tag for the current deterministic model.
  • Fallback path If mnemonic metadata is unavailable, the code still supports a deterministic fallback derived from Falcon-root material for bundle creation.

Restore Limits That Matter

  • Legacy random bundles If an old wallet used random private bundle generation, mnemonic alone cannot recreate the original Kyber secret for historical private receives.
  • Historical note recovery Without the old Kyber secret or preserved bundle artifacts, public funds can restore while old private outputs targeted to that legacy bundle remain undecryptable.
  • Advanced lockbox mode If the wallet exported an external Kyber unlock file, that file still belongs in the recovery set alongside the password and normal wallet backup.
