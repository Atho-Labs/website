Wallet Security + Recovery Payment-Only Model

Wallet custody built for payments, privacy, and recovery.

The Atho wallet is aligned to the current chain design: public payment addresses, private bundle identities, encrypted local custody, and deterministic recovery. It stays focused on sending, receiving, and restoring funds across the public and private lanes.

Wallet Policy Snapshot

  • Encryption stack Argon2id KDF, AES-256-GCM payload encryption, AES wrap, and Kyber-based DEK wrap.
  • Address surface Regular payment addresses plus private bundle / X-address material for private receive flows.
  • Fee policy 1 atom/vB floor, 500 atoms minimum transaction fee, and 1 atom dust limit.
  • Spendability 6 standard confirmations, 6 private confirmations, and 150 block coinbase maturity.
Wallet Addressing + Fees

Public payment addresses stay simple while private receive flows use bundle identity.

Atho’s active wallet surface has two user-facing destination types: a regular payment address for public value and a private bundle / X-address for Platinum Shield transactions.

Public Payment Address Pipeline

Public ownership is now regular-only. The wallet derives one payment address class from Falcon public-key material.

Address Protection and Key Isolation

The wallet keeps the active public payment surface narrow while still isolating private receive material for private transaction workflows.

  • Payment-only public ownership Consensus recognizes regular payment addresses only; legacy bond/stake address roles are not active.
  • Visible vs encrypted fields Base56 address and identifier data stay shareable; mnemonic, private keys, and raw secret material remain encrypted while locked.
  • Fail-closed lock policy Sensitive actions refuse to run until wallet auth requirements are satisfied.
  • Atomic local writes Wallet files use lock-protected and backup-safe write behavior to reduce corruption risk.

Private Transaction UX in Wallet Nodes

Wallet nodes assemble private transactions locally: sender auth, note selection, bundle parsing, X-address binding, payload encryption, and deterministic witness context all happen before mempool submission.

  • Public -> private funding Spend public UTXOs and create private commitments targeted to the recipient bundle.
  • Private -> private transfer Consume private notes via nullifiers and create fresh private outputs without exposing public value flow.
  • Private -> public release Return value to a regular public payment address when transparent settlement is required.

Recipient Bundles, QR, and Binding Checks

  • Reusable receive bundles Wallets can expose reusable private receive data while keeping note derivation deterministic and counter-based.
  • QR and copy workflows Private receive screens can share both bundle payload and derived X-address for low-friction initiation.
  • Fail-closed binding The wallet rejects mismatched bundle/X-address combinations before the private send path is allowed to continue.
  • Password-gated private sends Private actions require unlock policy checks on each sensitive send path.

Mnemonic Recovery Options at Wallet Creation

Wallet creation remains mnemonic-first. The current flow supports 12, 24, or 48 words and deterministically rebuilds the same payment and private receive identity when the same passphrase and derivation path are used.

12 Words 24 Words (Default) 48 Words
  • Creation options Generate new mnemonic wallet, recover from existing mnemonic, or import an existing Falcon key export.
  • Passphrase handling Passphrases are not stored in exports; keeping them separately is part of safe recovery hygiene.
