# SHA3-384, HPK, and Addressing

Date: 2026-04-04

This document describes how Atho uses SHA3-384 for identity and transaction hashing.

## 1) Hash Primitives
Core implementation:
- `Src/Utility/hash.py`

Primary primitives:
- SHA3-384 for canonical tx/block identity workflows.
- SHA3-256 for checksum/helper cases (for example address checksum paths).

## 2) HPK (Hashed Public Key)
- HPK is derived from public key bytes using SHA3-384.
- Raw digest size: `48 bytes`.
- Hex representation: `96 hex chars`.

Internal destination binding uses HPK form (`ATHO`/`ATHT` tagged form in many paths).

## 3) Public Key Size Policy (Falcon)
Current policy in constants:
- Canonical Falcon public key length: `897 bytes`.
- Optional legacy acceptance path can allow `1024` byte format during migration.

HPK derivation is always based on raw bytes, regardless of display encoding.

## 4) Base56 Address Layer
Base56 is user-facing representation on top of HPK.

Relevant code:
- `Src/Blockchain/base56.py`

Behavior:
- decode Base56 -> canonical HPK binding,
- encode HPK -> Base56 with checksum,
- keep display-friendly address format while internal flows stay hash-based.

## 5) Transaction Hashing Roles
- `txid`: no-witness deterministic hash path.
- `wtxid`: full-transaction hash including witness.
- Signing digest: deterministic no-witness signing body digest.

Relevant code:
- `Src/Utility/txdhash.py`
- `Src/Transactions/tx.py`
- `Src/Transactions/txvalidation.py`

## 6) Binary Integrity Pinning
SHA3-384 is also used for native binary integrity metadata and pin checks.

Related files:
- `Binaries/pin_registry.json`
- `Binaries/<platform-tag>/binary_meta.json`
- `Src/Utility/binarypin.py`
- `Src/Accounts/falconcli.py`

## 7) Operational Notes
- Keep address normalization centralized via existing helper paths.
- Keep user-facing Base56 stable; do not alter internal HPK digest length.
- Keep binary pin metadata refreshed whenever binaries are rebuilt.
